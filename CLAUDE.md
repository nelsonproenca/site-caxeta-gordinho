## Commands

- `npm run dev` — start the dev server.
- `npm run build` — production build.
- `npm run start` — run the production build.
- `npm run lint` — ESLint (flat config, `eslint.config.mjs`).
- `supabase db push` — apply pending `supabase/migrations/*.sql` to the linked remote project (no local Docker/Postgres in this environment — there is no `supabase start` workflow here, migrations are written and pushed directly to the real linked project).
- `supabase gen types typescript --linked > lib/supabase/database.types.ts` — regenerate types after any migration; the Supabase clients in `lib/supabase/{client,server,service}.ts` are all typed with `Database` from this file, so a schema change without regenerating will silently type-check against a stale schema.
- `node --env-file=.env.local --import tsx scripts/smoke-test.ts` — exercises the Fase 1 data flow (account creation, scoring, live session, match results, ranking math) and a light RLS cross-tenant/anon check, directly against the real linked project, then cleans up after itself. Run after any RLS policy or scoring-math change.
- `npm run test:rls` — Vitest RLS isolation suite (`tests/rls.test.ts`), the one the engineering plan calls out as a prerequisite before starting Fase 2 (M1, risk #1). Proves an admin/moderator of one `tiktok_account` can never read or write another account's protected rows, across every RLS-protected table, plus role gating (owner vs moderator) and the tables with no direct client write (`tiktok_accounts`, `players`). Also runs against the real linked project (no local Postgres here — see below), creates and tears down its own throwaway accounts/admins/players each run. Run after any RLS policy change; extend it when a new table gets a `has_account_access`/`is_account_owner` policy. **Does not yet cover `caxetao_events`/`caxetao_registrations`** — see Fase 2 section below.
- `npm run storybook` — dev server (port 6006) for the component library (`.storybook/`, stories co-located as `*.stories.tsx` next to each component under `components/`). `npm run build-storybook` produces a static build in `storybook-static/` (gitignored). Framework is `@storybook/nextjs-vite` (Vite-based, not a real Next.js server) — stories are limited to presentational components; anything that imports a `"use server"` action (`PlayerForm`, `CaxetaoRegisterForm`, `RegisterPlayerForm`, `CreateEventForm`, ...) is deliberately **not** storied, since Vite doesn't run the Next.js RSC compiler that makes `"use server"` work — importing one would very likely break the build. `.storybook/preview.tsx` imports `app/globals.css` directly (Storybook never renders `app/layout.tsx`, so this is the only way the design tokens/utility classes reach the canvas) and `.storybook/preview-head.html` loads the same Google Fonts `next/font` would otherwise inject, for visual fidelity. The scaffolded `@storybook/addon-vitest` integration (component tests via a headless Chromium/Playwright, wired into `vitest.config.ts` as a second `projects` entry) was removed right after `storybook init` — it would have made `npm run test:rls` also spin up a browser and run every story as a test, well past what "a Storybook to look at the design system" asked for.

## Deploy (Vercel)

Standard Next.js App Router project — Vercel auto-detects the framework, runs `npm run build`, and needs no `vercel.json`. Node `>=20.9.0` is pinned in `package.json`'s `engines` (Next 16's own peer requirement) so a project-level Node version override on Vercel picks something compatible.

- **Env vars** (Vercel project → Settings → Environment Variables): the four in `.env.local.example` — `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY` (server-only, never expose to a `NEXT_PUBLIC_*` var or client bundle), `NEXT_PUBLIC_SITE_URL`. Set `NEXT_PUBLIC_SITE_URL` to the real production domain, **not** `http://localhost:3000` — it's the only thing `metadataBase` (`app/layout.tsx`) and the magic-link/signup email redirect (`lib/actions/auth.ts` → `${NEXT_PUBLIC_SITE_URL}/auth/confirm`) have to go on.
- **Supabase Auth → URL Configuration**: add the production `NEXT_PUBLIC_SITE_URL` (and its `/auth/confirm` path) to the project's allowed Redirect URLs in the Supabase dashboard — this is external config, not something a code change here can do. Without it, admin login magic links/signup confirmations will redirect to a rejected URL.
- **Migrations are not part of the build.** `supabase db push` is a separate, manual step (see Commands above) — deploying a commit that assumes a new migration is applied does nothing to actually apply it. Push pending migrations (and regenerate `lib/supabase/database.types.ts`) before or alongside deploying code that depends on them, not after.
- **Storybook is not part of `npm run build`** — Vercel's default build only runs the Next.js app, so its dependencies/config don't affect the deployed site. If you also want Storybook hosted (e.g. a separate `storybook.<domain>` project), that's a second Vercel project pointed at this same repo with build command `npm run build-storybook` and output directory `storybook-static`.

## What this project is

A platform to manage a TikTok creator's live "Caxeta" (card game) streams: manual scoring of matches per live session, weekly/season rankings, a Saturday "Caxetão" event with configurable registration (principal/substitute slots, closing by time or by count), World-Cup-style championships (groups → knockout), and multi-account management (one platform, several managed TikTok accounts, admins/moderators scoped per account).

Full domain rules, entities, and the proposed Postgres schema are in `prd.md` §4–§7 — do not re-derive these from scratch, reference them.

## Key architectural constraints (non-obvious, drives everything)

- **No TikTok API access for live/follower data.** There is no official API to read live-room participants or a creator's follower list. All match results are entered manually by an admin/moderator through the panel during/after the live. "Active/most engaged followers" are computed from in-platform activity (matches played, live participations, Caxetão/championship participation) — never fetched from TikTok. See `prd.md` §1.1.
- **Players are global, participation is per-account.** A `players` row is one person across the whole platform; their scores, live participations, Caxetão registrations, and championship history are always scoped to a `tiktok_account_id`. Any query/feature touching scores or rankings must filter by the active managed account.
- **Players have no password-based auth** (lightweight signup: display name + unique TikTok handle, optional WhatsApp). Only admins/moderators authenticate (Supabase Auth). Don't build a full auth flow for players in the MVP.
- **Scoring rules are per-account and admin-configurable**, not hardcoded constants — points for each result type (e.g. "vitória lambreta", "derrota normal") live in a `scoring_rules` table and are editable without a code change. `match_results.points_awarded` is a snapshot at write time, so historical totals must never be recalculated from current `scoring_rules` values.
- **Multi-tenancy by TikTok account**: an admin can own/moderate several `tiktok_accounts`; access is governed by `admin_account_access` (role: owner/moderator per account). Authorization (Supabase RLS) should key off this join table, not off a single global admin role.

Next.js 16 (App Router, TypeScript) + Supabase (Postgres + Auth + RLS) + Tailwind v4, deployed on Vercel. ## Fase 1 (MVP) — implemented

Migrations `supabase/migrations/2026071500000{1..11}_*.sql` create `admins` (1:1 with `auth.users`), `tiktok_accounts`, `admin_account_access`, `players`, `scoring_rules`, `score_periods`, `live_sessions`, `live_participants`, `matches` (with `tiktok_account_id` denormalized — deliberate deviation from prd.md §7 to keep every write policy a single `has_account_access()` check instead of a join), and `match_results`. Two SECURITY DEFINER helper functions (`has_account_access`, `is_account_owner`) back nearly every RLS policy — reuse them rather than re-writing the `admin_account_access` exists-subquery. Accounts are created exclusively through the `create_tiktok_account` RPC (also seeds the 4 default `scoring_rules`), because `tiktok_accounts` has no direct insert policy — the first `admin_account_access` row for a new account can't otherwise satisfy `is_account_owner()`.

Later migrations (`2026071700001{3,4}_*.sql`): `score_periods` gained an open/closed `status` (unique partial index caps it at one open period per account) and `live_sessions`/`matches` gained a denormalized `score_period_id`, copied from the live session onto every match via a `BEFORE INSERT` trigger — mirrors the `tiktok_account_id` denormalization so a period's ranking is a single-level join (`lib/scoring/get-ranking.ts` → `getPeriodRanking`). **Gotcha this created and fixed**: any table that denormalizes `tiktok_account_id` (or similar) purely for RLS speed is a spoofing risk — `has_account_access(tiktok_account_id)` only validates the column value on the row being written, not that it actually agrees with a related row's real account (e.g. `matches.live_session_id` could point at a different account's live session while `tiktok_account_id` still names an account the caller owns). `matches` now has a second trigger (`matches_validate_account`) that raises if the two disagree — replicate that pattern for any future denormalized-account-id table instead of trusting RLS alone. Admin UI for all this lives at `/admin/[accountId]/ranking` (current period + podium + close action) and `/admin/[accountId]/ranking/[periodId]` (historical periods); periods are created manually by the admin (label typed by them), never auto-opened.

`players` has no insert/update policy for anon or authenticated at all — both public self-registration and admin "quick add" during a live go through Server Actions using `lib/supabase/service.ts` (service-role, bypasses RLS), so handle normalization/dedupe lives in one place. Every other admin-facing write goes through the regular per-request client (`lib/supabase/server.ts`) and relies on RLS, not application-level role checks.

Admin routes live under `app/admin/(shell)/` (route group, doesn't affect the URL) so `app/admin/login` can render without the sidenav shell that the rest of `/admin/**` gets from `app/admin/(shell)/layout.tsx`. `proxy.ts` redirects unauthenticated requests to `/admin/login` for anything under `/admin` except that route.

`lib/scoring/get-ranking.ts` is the single source of truth for ranking math (points desc → count of positive-point results desc → handle alphabetical). The exact "vitórias lambreta"-specific tie-break from prd.md §12 is still an open decision (scoring_rules has no category column to distinguish a lambreta win from a normal one) — don't invent a category column without checking that decision first.

## Fase 2 (Caxetão) — implemented

Migrations `supabase/migrations/2026071700001{5,6,7}_*.sql` add `caxetao_events` and `caxetao_registrations`, then extend `matches` with a `caxetao_event_id` column so a match can belong to either a live session or a Caxetão event (`matches_exactly_one_source` check constraint), extending `matches_validate_account`/`matches_set_score_period` accordingly — Caxetão matches deliberately get no `score_period_id`, since Caxetão is a separate competition from the weekly/season ranking. `caxetao_registrations` has no insert policy for anon/authenticated at all (same reasoning as `players`): both self-registration and admin manual add funnel through `registerPlayerForEvent` (`lib/caxetao.ts`) on the service-role client, which is the one place that assigns principal-vs-substitute and FIFO `queue_position`. Status *updates* after creation (presence, cancellation, promotion) go through the regular per-request client instead, since admin already has direct RLS write access there and there's no cross-row math to protect.

`lib/caxetao.ts` also has `syncCaxetaoEventStatus` — lazily flips `scheduled → registrations_open → registrations_closed` against the wall clock (same "no cron in this stack" pattern as `closeStaleLiveSessions`), called wherever a single event is read before acting on it. **Gotcha this created and fixed**: `registerPlayerForEvent` originally closed the event (`status = 'registrations_closed'`) as soon as `max_principals` was reached — but prd.md §4.5 says players beyond that cap should keep signing up as substitutes (up to `max_substitutes`), so reaching the principal cap must *not* stop registration outright. It now only auto-closes once principals **and** substitutes are both at capacity; an event with an uncapped substitute queue (`max_substitutes` null) is therefore never auto-closed by count, only by the time deadline or an admin. The three pt-BR label maps (`CAXETAO_STATUS_LABEL`, `CAXETAO_STATUS_VARIANT`, `CAXETAO_CLOSE_RULE_LABEL`) live in this same file — every page that renders an event's status/close_rule imports them rather than redefining its own copy.

The admin "Novo Caxetão" form (`create-event-form.tsx`) only offers `close_rule` **time** or **count** as a two-button toggle (no default selected, submit disabled until one is picked) — `both` is still a valid DB value and still handled by `createCaxetaoEvent`/`registerPlayerForEvent`, it's just not reachable from this form; a deliberate scope-simplification, not a bug, if you're wondering why the UI only shows two options where the schema allows three. The two field groups (time window vs. capacity) are mutually exclusive and always render on their own row, never mixed with `Data do evento`/`Regra de encerramento`. Picking "Por quantidade" submits `registration_opens_at` as "now" (São Paulo wall time, via `nowInSaoPauloAsDatetimeLocal`) through a hidden input instead of asking the admin to schedule an opening — that mode has no deadline for an opening time to be scheduled against.

Admin UI lives at `/admin/[accountId]/caxetao` (list + create) and `/admin/[accountId]/caxetao/[eventId]` (principals/substitutes, manual add via modal, cancel/no-show, FIFO promotion, `Iniciar evento`/`Finalizar evento` stage transitions). Both the list and detail cards follow a title/subtitle/plain-text hierarchy: `Data · Regra` in `font-display` as the title, rule-specific values as an `.text-ink-dim` subtitle, and — only while registration is open — a third plain-text line with either a live countdown (`components/closing-countdown.tsx`, `ClosingCountdown`) or "Faltam N jogadores para encerrar as inscrições".

**Player view (public, no auth)**: `app/(public)/[accountHandle]/layout.tsx` wraps `ranking`/`caxetao`/`inscricao` in a shared tab nav (`components/ui/tab-bar.tsx` — also used by the admin account shell, see below). `/[accountHandle]/caxetao` shows the same principals/substitutes lists as admin, a `.card-stat`-styled (big number) countdown-or-remaining-slots readout for extra visibility, and self-registration via a modal (`caxetao-register-modal.tsx`) instead of an inline form. `RankingBoard` (`components/ranking-board.tsx`, shared by admin and public ranking pages) got a search box that filters by name/@handle while still showing each match's real rank, not its index in the filtered list. The root `/` page is the player entry point — it lists active `tiktok_accounts` linking into their `/ranking`, with `/admin/login` demoted to a small link at the bottom instead of being the page's main call to action.

`components/ui/modal.tsx` is a `<dialog>`-backed modal shared by every "open a form in a modal" flow (`register-player-modal.tsx`, `caxetao-register-modal.tsx`) — it renders the `<dialog>` itself as an invisible full-viewport flex-centering frame with the visible card as a nested `.modal-content` div. **Gotcha this created and fixed**: don't center a `<dialog>` by relying on the UA stylesheet's `dialog:modal { margin: auto }` — Tailwind v4's Preflight resets `margin: 0` in `@layer base`, and even though that's a *lower*-priority layer than unlayered author CSS, it's still higher priority than the UA stylesheet's own (lowest) tier, so Preflight's reset silently wins and the dialog collapses to the top-left corner. Center with flex (`.modal-overlay[open]{ display:flex; align-items:center; justify-content:center; }`) instead — it has no dependency on margin at all. `ModalHeader` (title + close-✕ button) is exported from the same file since it was identical boilerplate at every modal call site.

**Known gap**: `tests/rls.test.ts` does not yet cover `caxetao_events`/`caxetao_registrations` — deferred to the end of the phase per explicit instruction rather than extended after each migration (unlike the stated convention above). Extend it before starting Fase 3.
